# @workkit/workflow Implementation Plan — "Durable Agent Execution"

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build durable, checkpoint-based workflow execution on Cloudflare Workers. Step chains with type inference, crash recovery via replay, saga compensation.

**Architecture:** Builder pattern accumulates typed step chain → `build()` produces `WorkflowDef` → `run()` creates a DO per execution → steps execute sequentially with journal checkpointing → crash recovery replays from last checkpoint.

**Tech Stack:** TypeScript, @workkit/do (typed storage, alarms), @workkit/errors (retry strategies), Vitest, bunup

**Spec:** `workkit/docs/superpowers/specs/2026-03-24-workflow-design.md`

### Task 4: Step Journal

The journal is the persistence layer — records each step's state in DO storage.

- `createJournal(storage)` — factory
- `writeEntry(index, entry)` — put to `wf:step:{index}`
- `readEntry(index)` — get from `wf:step:{index}`
- `readAll()` — read all step entries
- `getStepCount()` — read `wf:step:count`
- `setStepCount(n)` — write `wf:step:count`

Each entry: `{ index, name, status, input, output?, error?, startedAt?, completedAt?, duration?, attempt, idempotencyKey? }`

Commit: `feat(workflow): add step journal`

### Task 5: Workflow Builder

The builder accumulates steps with recursive type inference. This is the DX layer.

- `createDurableWorkflow(name, config)` — returns `WorkflowBuilder<TInput, {}>`
- `.step(name, handler, options?)` — adds a step, returns new builder with accumulated types
- `.sleep(duration)` — adds a sleep step
- `.wait(name, eventName, options?)` — adds a wait step
- `.retry(strategy)` — returns `RestrictedWorkflowBuilder`
- `.timeout(duration)` — returns `RestrictedWorkflowBuilder`
- `.onFailure(handler)` — finalizes with compensation, returns `WorkflowDef`
- `.build()` — finalizes without compensation, returns `WorkflowDef`

Internally stores steps as an array of `StepDefinition` objects. `build()` creates a `WorkflowDef` by passing the step array + config to `createWorkflowDef()`.

Commit: `feat(workflow): add workflow builder`

### Task 6: Step Executor

The executor runs steps sequentially with retry, timeout, and journal checkpointing.

- `createExecutor(steps, config, storage, journal)` — factory
- `execute(input)`:
  1. Write execution metadata to storage
  2. For each step:
     a. Check journal — if already completed, use cached output
     b. If pending/running: execute handler with retry + timeout
     c. Write journal entry (status, output/error, timing)
     d. If failed after retries: stop, trigger compensation
  3. Write final status (completed/failed)
- Retry: uses `@workkit/errors` `getRetryDelay()` with configurable strategy
- Timeout: per-step, uses AbortSignal
- Crash recovery: on re-execution, reads journal, skips completed steps, resumes from first non-completed

Commit: `feat(workflow): add step executor`

### Task 7: Compensation (Saga Pattern)

- `runCompensation(handler, context)` — executes the compensation handler
- Context includes: input, failedStep, error, completedSteps, stepOutputs, logger
- Compensation errors are logged but don't fail the workflow (best-effort)

Commit: `feat(workflow): add saga compensation`

---

### Task 8: Workflow Execution DO

The DO class wraps the executor with HTTP interface.

- `WorkflowExecutionDO` implements `DurableObject`
- `fetch(request)` routes:
  - `POST /execute` — start execution with input + step definitions
  - `GET /status` — return execution metadata
  - `GET /journal` — return step journal
  - `POST /cancel` — cancel execution
  - `POST /resume` — resume from wait step with event payload
- Uses `createExecutor()` internally

Commit: `feat(workflow): add WorkflowExecutionDO`

---

### Task 9: WorkflowDef

The finalized workflow definition with `run()`, `cancel()`, `execution()`.

- `createWorkflowDef(name, steps, config)` — factory
- `.run(input, options?)`:
  1. Generate execution ID
  2. Create DO instance
  3. POST /execute with input + serialized steps
  4. Return `ExecutionHandle`
- `.cancel(executionId)` — POST /cancel to DO
- `.execution(executionId)` — returns handle for existing execution
- ExecutionHandle: `status()`, `result()`, `meta()`, `journal()`, `resume()`, `cancel()`

Commit: `feat(workflow): add WorkflowDef`
